#+title:     Error Handling Guidelines
#+language:  en

* Philosophy

This project uses *semantic error types* with ~thiserror~ rather than
generic error wrapping with ~anyhow~.

** Why Not ~anyhow~?

While ~anyhow~ is convenient, it encourages lazy error handling that
loses important context.  Consider:

#+begin_src rust :exports code
// BAD - Using anyhow blindly
fn load_config() -> anyhow::Result<Config> {
    // What file?  Why did it fail?
    let contents = std::fs::read_to_string("config.toml")?;
    // What was wrong with the TOML?
    let config = toml::from_str(&contents)?;
    Ok(config)
}
#+end_src

When this fails, you get a generic error like "No such file or
directory" with no context about:

- *What* file was being read.
- *When* in the application lifecycle this happened (startup?
  runtime?).
- *Why* it matters (configuration is required for startup).

This is no better than a Python program that blows up with a stack
trace.

* Core Principle: Always Add Context

*All errors must include sufficient context to diagnose the problem
without access to the source code or the running environment.*

** Why Context Matters

When errors occur in production (or are reported by users who cannot
share their environment), you often only have access to the error
message.  Error messages without context make debugging difficult or
impossible.

*** Bad: Missing context

#+begin_src rust :exports code
let content = std::fs::read_to_string(path)
  .map_err(|e| {
    error!("Failed to read file: {}", e);
    e
  })?;
#+end_src

Error message: ~Failed to read file: No such file or directory (os
error 2)~

Questions this leaves unanswered:
- What file was it trying to read?
- What was the current working directory?

*** Good: Includes full context

#+begin_src rust :exports code
let content = std::fs::read_to_string(&path)
  .map_err(|e| {
    error!(path = %path, "Failed to read file: {}", e);
    e
  })?;
#+end_src

Error message includes the path and any relevant identifiers.

* Error Type Design with ~thiserror~

Use semantic error types that explain what went wrong and why it
matters:

#+begin_src rust :exports code
use thiserror::Error;

#[derive(Debug, Error)]
pub enum FlakeError {
  #[error(
    "Failed to spawn nix for attribute {attr} at flake {flake}: {source}"
  )]
  Spawn {
    flake: String,
    attr: String,
    #[source]
    source: std::io::Error,
  },

  #[error("Failed to parse host list from flake {flake}: {source}")]
  ParseHostList {
    flake: String,
    #[source]
    source: serde_json::Error,
  },
}
#+end_src

Now when it fails, you get:

#+begin_example
Failed to spawn nix for attribute .#nixosConfigurations at flake .:
No such file or directory (os error 2)
#+end_example

This tells you:
- *What*: Spawning nix failed.
- *Where*: The specific attribute and flake.
- *Why*: The underlying OS error.

** Error Type Hierarchy

Structure your error types in layers:

1. *Low-level errors* (in modules) — specific to that component.
2. *Top-level errors* (in main.rs) — application lifecycle errors.

#+begin_example
AppError::FlakeQuery
    └─ FlakeError::Spawn
        └─ std::io::Error (No such file or directory)
#+end_example

Each layer adds context about what the application was trying to do.

** When ~anyhow~ Might Be Acceptable

The only time ~anyhow~ is acceptable is for *quick prototypes* or
*throwaway scripts*.  Once you know what errors can occur, convert to
semantic error types.

* Guidelines

** Every Error Variant Should Explain Context

- What operation failed.
- Why it matters.
- Include relevant data (host names, attributes, flake paths, etc.).

** Use ~#[source]~ for Error Chains

- Preserves the full error chain.
- Allows inspection of root causes.

** Be Specific in Error Messages

Bad example:

#+begin_src rust :exports code
#[error("Nix failed: {0}")]
NixError(std::io::Error),
#+end_src

Good example:

#+begin_src rust :exports code
#[error(
  "Failed to spawn nix for attribute {attr} at flake {flake}: {source}"
)]
Spawn {
  flake: String,
  attr: String,
  #[source]
  source: std::io::Error,
},
#+end_src

** Handle Errors at the Right Level

- Don't catch errors too early (loses context).
- Don't let errors bubble up without adding context.
- Transform errors to add domain-specific context.
